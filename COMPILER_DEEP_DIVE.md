# 📖 LughatDad — Compiler Deep Dive
> **For exam preparation**: Everything the doctor may ask about Lexical Analysis, Parsing, Interpretation, and the full compiler pipeline.

---

## 1. What is a Compiler vs. Interpreter?

| Term | Definition | LughatDad's case |
|------|-----------|-----------------|
| **Compiler** | Translates source → target language (e.g. machine code) at once | We use **JavaCC** to generate a Java parser from a grammar spec |
| **Interpreter** | Executes source code directly, statement by statement | LughatDad **interprets** Arabic programs at runtime (tree-walking) |
| **Transpiler** | Compiles to another high-level language | Not used here |

> LughatDad is technically an **interpreted language** whose interpreter is built with **compiler construction tools** (JavaCC).

## 2. Phase 1 — Lexical Analysis (Lexer / Tokenizer / Scanner)

### What is it?
The lexer reads raw source characters and groups them into **tokens** — the smallest meaningful units of the language.

### How LughatDad does it
Defined in `LughatDad.jj` using JavaCC's **TOKEN** and **SKIP** rules:

```
SKIP : { " " | "\t" | "\n" | "\r" }         // ignore whitespace
SKIP : { < "//" (~["\n","\r"])* > }          // ignore comments

TOKEN: { <MUTA_GHAYYER: "\u0645\u062a\u063a\u064a\u0631"> }  // متغير
TOKEN: { <ITBA3:        "\u0627\u0637\u0628\u0639"> }         // اطبع
TOKEN: { <ETHA:         "\u0625\u0630\u0627"> }               // إذا
TOKEN: { <WA_ELLA:      "\u0648\u0625\u0644\u0627"> }         // وإلا
TOKEN: { <BAYNAMA:      "\u0628\u064a\u0646\u0645\u0627"> }   // بينما
TOKEN: { <SAH:          "\u0635\u062d"> }                     // صح
TOKEN: { <KHATA:        "\u062e\u0637\u0623"> }               // خطأ

TOKEN: { <ARABIC_IDENTIFIER: <ARABIC_LETTER>(<ARABIC_LETTER>|<DIGIT>)*> }
TOKEN: { <NUMBER: (<DIGIT>)+ ("." (<DIGIT>)+)? > }
TOKEN: { <STRING: "\"" (~["\""])* "\""> }
TOKEN: { <PLUS:"+"> | <MINUS:"-"> | <MULTIPLY:"*"> | <DIVIDE:"/"> }
TOKEN: { <EQ:"=="> | <NEQ:"!="> | <GT:">"> | <LT:"<"> | <GTE:">="> | <LTE:"<="> }
TOKEN: { <ASSIGN:"="> | <LPAREN:"("> | <RPAREN:")"> }
TOKEN: { <LBRACE:"{"> | <RBRACE:"}"> | <SEMICOLON:";"| "\u061B"> }
```

### JavaCC generates from these rules:
- **`LughatDadTokenManager.java`** — the actual lexer (DFA-based state machine)
- **`LughatDadConstants.java`** — integer IDs for each token type
- **`Token.java`** — a token object with `.image` (matched text) and `.kind` (type ID)
- **`TokenMgrError.java`** — thrown when a character matches no token rule

### Key design decisions:
1. **Unicode codepoints** are used for Arabic keywords (not raw Arabic text) to ensure encoding safety
2. **ARABIC_IDENTIFIER** uses range `\u0620–\u065F` — covers all Arabic letters
3. **Arabic digits** (٠–٩, U+0660–U+0669) are supported in the DIGIT fragment
4. Arabic semicolon `؛` (U+061B) is accepted alongside standard `;`

## 3. Phase 2 — Parsing (Syntax Analysis)

### What is it?
The parser takes the token stream from the lexer and checks if it conforms to the **grammar rules** of the language. It builds an **AST (Abstract Syntax Tree)**.

### Grammar Formalism: BNF / EBNF
JavaCC uses **EBNF** (Extended Backus-Naur Form). Our grammar in plain BNF:

```
program     → statement* EOF
statement   → declaration | assignment | printStmt | ifStmt | whileStmt
declaration → متغير IDENTIFIER = expression ;
assignment  → IDENTIFIER = expression ;
printStmt   → اطبع ( expression ) ;
ifStmt      → إذا ( condition ) block [ وإلا block ]
whileStmt   → بينما ( condition ) block
block       → { statement* }
condition   → expression [ relop expression ]
expression  → term ( (+ | -) term )*
term        → unary ( (* | /) unary )*
unary       → [-] primary
primary     → NUMBER | STRING | IDENTIFIER | ( expression ) | صح | خطأ
relop       → > | < | == | != | >= | <=
```

### How JavaCC implements it:
Each grammar rule becomes a **Java method** in `LughatDad.java` (auto-generated):
```java
BlockStmt program()   { ... }   // top-level
Stmt statement()      { ... }   // dispatcher
Stmt declaration()    { ... }   // متغير x = expr;
Stmt ifStmt()         { ... }   // إذا (cond) block [وإلا block]
Expr expression()     { ... }   // handles + and -
Expr term()           { ... }   // handles * and /
Expr primary()        { ... }   // atoms: numbers, strings, vars
```

### Parser type: LL(1) / LL(k)
LughatDad uses a **top-down recursive descent LL(1) parser** (generated by JavaCC). It reads tokens **left-to-right** and applies **leftmost derivations** with **1 token lookahead**. The `LOOKAHEAD(1)` directive is used for the optional `وإلا` branch.

### What JavaCC generates:
- **`LughatDad.java`** — the parser (all grammar rules as Java methods)
- **`ParseException.java`** — thrown on syntax errors with helpful messages
- **`SimpleCharStream.java`** — character stream reader feeding the lexer

## 4. Phase 3 — Interpretation (Semantic Actions + Tree Walking)

### What is it?
Instead of generating code, LughatDad **directly executes** the AST. This is a **tree-walking interpreter**.

### AST Node Classes (embedded in LughatDad.jj)

**Expression nodes** (subclass `Expr`, have `eval()` method):
```java
NumExpr(double val)           // number literal: 42, 3.14
StrExpr(String val)           // string literal: "مرحبا"
BoolExpr(boolean val)         // صح / خطأ
VarExpr(String name)          // variable lookup in symbolTable
BinOp(Expr l, String op, Expr r)  // arithmetic & comparison
```

**Statement nodes** (subclass `Stmt`, have `execute()` method):
```java
DeclStmt(String name, Expr e)    // متغير x = expr → put in symbolTable
AssignStmt(String name, Expr e)  // x = expr → update symbolTable
PrintStmt(Expr e)                // اطبع(expr) → System.out.println
IfStmt(Expr c, Stmt t, Stmt e)   // conditional execution
WhileStmt(Expr c, Stmt body)     // loop
BlockStmt(List<Stmt>)            // list of statements
```

### Symbol Table
```java
static HashMap<String, Object> symbolTable = new HashMap<>();
```
- Stores variables as `String → Object` (dynamic typing)
- `Object` can be `Double`, `String`, or `Boolean`
- Declared with `DeclStmt`, updated with `AssignStmt`
- `VarExpr.eval()` throws `RuntimeException` if key not found

### How `BinOp.eval()` works:
```java
case "+": return d1 + d2;     // arithmetic
case "-": return d1 - d2;
case "*": return d1 * d2;
case "/": return d1 / d2;
case "==": return l.equals(r); // works for any type
case "!=": return !l.equals(r);
case ">":  return d1 > d2;    // comparison (cast to double first)
```

## 5. Full Execution Pipeline (One HTTP Request)

```
User types Arabic code in browser
         │
         ▼
POST /api/run  { "code": "متغير س = 5؛\nاطبع(س)؛" }
         │
         ▼
LughatDadServer.RunHandler
  - Extracts "code" from JSON body
  - Writes code to temp file: /tmp/lughatdad_XXXX.txt
         │
         ▼  spawns fresh JVM subprocess
java -cp /app/backend LughatDadRunner /tmp/lughatdad_XXXX.txt
         │
         ▼
LughatDadRunner.main()
  1. Resets symbolTable = new HashMap<>()
  2. Opens temp file as InputStream
  3. Creates parser: new LughatDad(inputStream)
         │
         ▼  LEXICAL ANALYSIS (LughatDadTokenManager)
  Characters → Tokens
  "متغير" → MUTA_GHAYYER
  "س"     → ARABIC_IDENTIFIER("س")
  "="     → ASSIGN
  "5"     → NUMBER("5")
  "؛"     → SEMICOLON
  "اطبع"  → ITBA3
  "("     → LPAREN
  "س"     → ARABIC_IDENTIFIER("س")
  ")"     → RPAREN
  "؛"     → SEMICOLON
         │
         ▼  PARSING (LughatDad.java)
  program()
    statement() → declaration()
      DeclStmt("س", NumExpr(5.0))
    statement() → printStmt()
      PrintStmt(VarExpr("س"))
  BlockStmt([DeclStmt, PrintStmt])
         │
         ▼  INTERPRETATION (execute/eval)
  DeclStmt.execute() → symbolTable.put("س", 5.0)
  PrintStmt.execute() → VarExpr.eval() → symbolTable.get("س") → 5.0
                      → System.out.println(5)
         │
         ▼
Output: "5" captured from subprocess stdout
         │
         ▼
JSON response: { "success": true, "output": ["5"], "error": null }
         │
         ▼
Browser displays: 5
```

## 6. Why Subprocess Isolation?

The `.jj` file uses `STATIC = true` — all parser state is in **static fields**:
```java
static HashMap<String, Object> symbolTable = new HashMap<>();
```

If the same JVM handled multiple requests, leftover state from one user's program would corrupt the next user's run. **Solution**: spawn a fresh `java` subprocess per request via `ProcessBuilder`. Each subprocess gets a clean JVM with empty static state. After execution, the process exits and all memory is freed.

## 7. Error Handling

| Error type | Where caught | What happens |
|-----------|-------------|-------------|
| Unknown character | `TokenMgrError` (lexer) | Subprocess exits with error message |
| Syntax error | `ParseException` (parser) | Arabic error: `خطأ نحوي: ...` |
| Undefined variable | `RuntimeException` (interpreter) | Arabic error: `خطأ: متغير غير معرّف` |
| Division by zero | `RuntimeException` | `خطأ: القسمة على صفر` |
| Timeout (>10s) | `LughatDadServer` | `خطأ: انتهت مهلة التنفيذ` |

## 9. Common Exam Questions & Answers

**Q: What is a token?**
A: The smallest meaningful unit in source code. Examples: keyword (`متغير`), identifier (`س`), number (`42`), operator (`+`), punctuation (`;`).

**Q: What is the difference between lexical and syntax error?**
A: Lexical error = unrecognized character (e.g. `@`). Syntax error = valid tokens in invalid order (e.g. `متغير = 5؛` — missing identifier).

**Q: What grammar type does LughatDad use?**
A: Context-Free Grammar (CFG), specifically LL(1) parsed top-down by recursive descent.

**Q: What is an AST?**
A: Abstract Syntax Tree — a tree representation of the program's structure. Each node is a statement or expression. LughatDad builds it implicitly by returning Java objects from parser methods.

**Q: What is the symbol table?**
A: A data structure (HashMap here) mapping variable names to their current values. Used by the interpreter to store and retrieve variable state during execution.

**Q: Why use JavaCC instead of writing a lexer/parser by hand?**
A: JavaCC generates correct, efficient lexers and parsers from a declarative grammar specification. It's faster and less error-prone than manual implementation.

**Q: What is LOOKAHEAD?**
A: JavaCC's mechanism for resolving ambiguity. `LOOKAHEAD(1)` means look at the next 1 token to decide which production to apply. Used in LughatDad for the optional `وإلا` branch in `ifStmt()`.
